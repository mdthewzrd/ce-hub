# ⚙️ Traderra — Technical Requirements & System Architecture

## 1. Overview
This document outlines Traderra’s technical architecture, deployment model, infrastructure requirements, observability, and scaling roadmap. It provides the full specification needed for engineering, DevOps, and AI runtime integration teams.

---

## 2. Architecture Summary

Traderra runs as a **modular distributed system**, optimized for AI-assisted analytics, journaling, and professional-grade performance tracking. The architecture prioritizes:
- Security-first multi-tenant design
- Low-latency AI context streaming
- Scalable ingestion (imports, brokers, APIs)
- Cloud-native reliability

### 2.1 Core Stack
| Layer | Tech | Purpose |
|--------|------|----------|
| Frontend | React + AGUI + Co-PilotKit | Adaptive GUI, AI context sync |
| API Gateway | FastAPI / Next.js | REST + WebSocket endpoints |
| AI Runtime | PydanticAI | Core orchestrator for Renata’s behavior |
| Database | PostgreSQL + pgvector | Persistent data and embeddings |
| Cache | Redis | Context cache + queues |
| Object Storage | R2 / S3 | Files, CSVs, exports, journal attachments |
| Market Data | Polygon API | Real-time and historical prices |
| Auth | Clerk | OAuth + session management |
| Billing | Stripe | Subscription + invoicing |
| Monitoring | OpenTelemetry + Grafana | Tracing and observability |

## 4. Infrastructure

### 4.1 Cloud Platform
**Baseline:** Vercel + Cloudflare Workers + AWS

| Service | Role |
|----------|------|
| **Vercel** | Host frontend + Next.js edge routes |
| **AWS ECS / Fargate** | Host FastAPI + PydanticAI runtime |
| **AWS RDS (PostgreSQL)** | Database cluster |
| **Cloudflare CDN** | Global cache for static assets |
| **AWS ElastiCache (Redis)** | Cache + queue backend |
| **Cloudflare KV** | Lightweight key/value for feature flags |
| **AWS S3 or Cloudflare R2** | File + journal storage |

### 4.2 Networking & Security
- All ingress through HTTPS/TLS 1.2+.  
- WAF protection via Cloudflare.  
- Private subnets for DB and cache.  
- Bastion-only SSH access.

### 4.3 Infrastructure as Code
- **Terraform** modules for AWS / Cloudflare provisioning.
- **Docker Compose** for local dev parity.

## 5. Deployment Model

### 5.1 Environments
| Environment | Purpose |
|--------------|----------|
| **Local** | Developer sandbox; uses SQLite + MinIO |
| **Staging** | Pre-prod test, runs same infra with feature flags |
| **Production** | Full HA deployment with multi-AZ support |

### 5.2 CI/CD Pipeline
- GitHub Actions (build → test → deploy).  
- Type checks (mypy), linting (ruff), unit + integration tests (pytest).  
- Auto migrations applied in staging, manual approval for prod.  
- Canary deploys for API workers.

## 6. Performance Targets

| Metric | Target | Description |
|---------|---------|-------------|
| **P50 latency** | <300ms | Standard API requests |
| **P95 latency** | <800ms | AI-assisted responses |
| **Context sync latency** | <200ms | AGUI → API → Renata round trip |
| **AI query throughput** | 1000 QPM | Horizontal scaling via workers |
| **Worker job duration** | <30s avg | Imports, analysis, review jobs |

### Scaling
- Horizontal scaling via ECS Fargate tasks and Redis queues.  
- Read replicas for analytics-heavy queries.  
- Worker autoscaling triggered by Redis queue depth.

## 7. Observability

### 7.1 Logging
- Structured JSON logs with `request_id` and `workspace_id` context.  
- Centralized via OpenSearch or Loki.

### 7.2 Tracing
- OpenTelemetry traces across frontend, API, AI runtime, and workers.  
- Trace IDs passed through request headers.  
- Instrumented libraries: `asyncpg`, `httpx`, `redis`, `pydanticai`.

### 7.3 Metrics
| Metric | Source | Target |
|---------|--------|--------|
| API latency | FastAPI | Prometheus |
| Queue depth | Redis | Grafana alerts |
| Error rates | API/Workers | Alerting if >2% |
| DB query time | Postgres | APM monitoring |

## 8. Reliability & Backups

### 8.1 Database
- Point-in-time recovery enabled.  
- Automated snapshots every 6 hours, retained 7 days.  
- Read replica in alternate AZ.

### 8.2 Object Storage
- Versioning enabled.  
- Cross-region replication for R2/S3.  
- Lifecycle rules: 90-day cold storage for attachments.

### 8.3 Redis
- AOF persistence.  
- Failover cluster (multi-node).  

### 8.4 Disaster Recovery
- RTO: <1 hour.  
- RPO: <15 minutes.  
- Recovery playbooks defined in `ops/docs/dr-procedures.md`.

---

## 9. Security

### 9.1 Data Protection
- TLS everywhere.  
- AES-256 encryption at rest.  
- Per-user encryption keys via AWS KMS.  
- Sensitive data (tokens, credentials) masked in logs.

### 9.2 Authentication & Authorization
- Clerk for identity and sessions.  
- JWT tokens signed + verified per request.  
- Role-based access enforced server-side via RLS.  

### 9.3 Network Security
- Private subnets for DB/cache.  
- No public DB endpoints.  
- Security groups allow only API/worker ingress.

### 9.4 Dependency Security
- Renovate bot for weekly dependency audits.  
- Snyk + GitHub Dependabot integration.

## 10. Scaling & Capacity Planning

| Component | Scaling Model | Threshold |
|------------|----------------|------------|
| API Workers | ECS autoscale | CPU > 70% for 5 min |
| Redis | Clustered | Queue depth > 1k |
| Database | Read replicas | QPS > 2k |
| Object Storage | Elastic | Automatic |
| AI Runtime | GPU scaling (if used) | Active sessions > 50 |

**Forecast:** MVP → 10k users, 100 concurrent Renata sessions, 1TB total trade data.

## 12. Deployment Flow Example

4. Canary 10% rollout → monitor metrics → full deployment.

## 13. Failover & Rollback

- Blue/green deployment strategy.  
- If metrics exceed error threshold, rollback via previous ECS task definition.  
- Database rollbacks handled via transaction snapshots.
