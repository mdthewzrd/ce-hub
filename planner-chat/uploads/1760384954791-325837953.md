# 🔐 Traderra — Security & Privacy Specification (v1)

## 1) Purpose & Scope
This document defines **security, privacy, and compliance** requirements for Traderra across infrastructure, application, data lifecycle, AI embeddings, vendors, and operations. It is implementation-grade and audited during each release.

---

## 2) Security Objectives
- **Tenant Isolation:** Strong workspace-level RLS and network isolation.
- **Data Minimization:** Collect only what is required for core features.
- **Defense-in-Depth:** Controls at network, host, app, and data layers.
- **Zero Trust by Default:** AuthN/AuthZ everywhere, least privilege.
- **Verifiable Compliance:** Artefacts for SOC2 readiness; auditability.

---

## 3) Data Classification & Inventory
| Class | Examples | Controls |
|------|----------|----------|
| **C1 – Public** | Marketing copy, docs | CDN cache; no auth |
| **C2 – Internal** | Feature flags, anon metrics | App auth; logging allowed |
| **C3 – Confidential** | Trades, journals, reviews, insights | RLS, at-rest encryption, audit logs |
| **C4 – Sensitive** | OAuth tokens, API keys, access tokens | KMS envelope encryption, vault, redaction, no logs |
| **C5 – PII** | Email, name, IP, device IDs | Consent, retention policy, DSR support |

**Inventory:** Maintain a living data map with table → fields → class → owner.

---

## 4) Identity, AuthN/Z & Tenancy
- **Identity Provider:** Clerk (OAuth, MFA, session management).
- **Auth:** JWT in `Authorization: Bearer <token>`; short-lived; refresh via Clerk.
- **Tenancy:** Row-Level Security (RLS) with `workspace_id` session variable.
- **Roles:** `owner`, `admin`, `member`, `viewer`. Server-side checks for mutating routes.
- **SCIM/SSO (Roadmap):** SAML/OIDC for enterprise; SCIM provisioning.
- **Session Security:** IP + UA binding optional; revoke on anomaly.

---

## 5) Secrets & Keys
- **KMS:** AWS KMS for envelope encryption; CMKs per environment.
- **Storage:** `integration_tokens` encrypted (AES-256-GCM). Never return full tokens via API.
- **Rotation:** Quarterly or on incident; automated rotation hooks for Notion/Drive/Brokers.
- **Transit:** TLS 1.2+ everywhere; HSTS; no plaintext protocols.
- **Development:** `.env` never committed; use SOPS or parameter store.

---

## 6) Network & Infrastructure Security
- **Perimeter:** Cloudflare WAF (bot filtering, rate limit, OWASP rules).
- **Segmentation:** Private subnets for DB/Redis; security groups allow only API/worker ingress.
- **Ingress:** HTTPS only; TLS termination at edge; mTLS optional for internal services.
- **Egress:** Egress allowlist for Polygon/Notion/Drive; SSRF protection on webhooks.
- **Container Security:** Read-only filesystem; non-root users; seccomp profiles.
- **Patching:** Weekly base image updates; automatic minor updates for OS/runtime.

---

## 7) Application Security
- **Framework:** FastAPI/Next.js hardened headers (CSP, XFO deny, XCTO, Referrer-Policy).
- **CSRF:** JWT + SameSite=strict cookies on web app; double-submit token for forms if needed.
- **XSS:** React auto-escaping; sanitize rich text; strict CSP (`script-src 'self'`).
- **SQLi:** Parameterized queries via ORM; RLS enforced.
- **RCE/Deserialization:** No dynamic eval; strict JSON schemas.
- **Rate Limiting:** Per-IP + per-user; exponential backoff on 429.
- **DoS Protections:** Queue backpressure; request body size caps; timeout budgets.

---

## 8) Logging, Auditing & Monitoring
- **Structured Logs:** JSON with `request_id`, `workspace_id`, and route. No secrets/PII in logs.
- **Audit Logs:** All mutations recorded: who/what/when/before/after + IP/UA.
- **Retention:** Logs 30–90 days; audit logs 180–365 days (configurable).
- **Observability:** OpenTelemetry traces; metrics (latency, errors, queue depth, cache hits).
- **Alerting:** Pager duty/on-call for P1 (auth outage, data leakage, API 5xx rate > 2%).

---

## 9) Privacy & Data Protection
- **Consent:** Explicit consent during onboarding for data processing and analytics.
- **Purpose Limitation:** Use data only for journaling/analytics/features user requested.
- **User Control:** Export (CSV/ZIP), delete account, revoke integrations from Settings.
- **DSRs:** Right of access, correction, deletion. SLA: 30 days.
- **Data Minimization:** Do not collect non-necessary attributes; optional fields opt-in.
- **International Transfers:** Data residency roadmap (US/EU regions); vendor DPAs.

---

## 10) Embeddings & AI Privacy
- **Scope:** Only journals, reviews, and anonymized context are embedded.
- **Redaction:** Strip PII (names, account numbers) before embedding; replace with placeholders.
- **Isolation:** Vectors scoped per workspace; no cross-tenant retrieval.
- **Encryption:** At rest encrypted; embeddings table not exposed to ad-hoc queries.
- **Model Usage:** No training on user data without explicit opt-in. Zero sharing with third parties beyond model inference provider under DPA.
- **Prompt Metadata:** Persist minimal context; no raw secrets or tokens in prompts.

---

## 11) Data Retention & Deletion
- **Default Retention:** Active data retained indefinitely while account active.
- **Soft Delete:** 30-day grace; recoverable by owner.
- **Hard Delete:** Purge trades, journals, embeddings, files; tombstone record retained for audit only (non-identifying).
- **Backups:** PITR; backups encrypted and rotated; purge deleted user data from backups via rolling window (where feasible) or document in policy.

---

## 12) Third-Party & Vendor Risk
- **Vendors:** Clerk, Stripe, Polygon, Notion, Google Drive, Cloudflare, AWS.
- **Due Diligence:** SOC2/ISO27001 reviews; DPAs; subprocessor disclosure.
- **Key Risks:** Token leakage, over-scoped permissions. Mitigation: least-privilege OAuth scopes, short-lived tokens.
- **Pen Tests:** Annual 3rd-party pen test; remediation tracked.

---

## 13) Webhooks & Integrations
- **Signing:** HMAC-SHA256 with shared secret; `X-Signature` header.
- **Replay Protection:** Nonce + timestamp ±5 minutes; store hashes of recent IDs.
- **Timeouts/Retry:** 3 retries with exponential backoff; idempotent handlers.
- **Scopes:** Only required scopes requested; revoke on disconnect.

---

## 14) Access Control & Admin Ops
- **Admin Console:** Guarded behind SSO + hardware key (WebAuthn). Read-only by default; write ops require break-glass workflow.
- **Break-Glass:** Time-bound elevated access with approval + audit; auto-revocation.
- **Least Privilege:** IAM roles per service; deny-by-default policies.
- **Key Management:** Rotate KMS keys annually; per-tenant keying (roadmap) for enterprise.

---

## 15) Backup, DR & Business Continuity
- **RPO/RTO:** 15 minutes / 1 hour.
- **Backups:** Automated snapshots; quarterly restore drills; checksum verification.
- **DR Plan:** Region failover runbook; infra-as-code for rapid redeploy.
- **Dependencies:** Document external SLAs (Polygon, Clerk, Stripe) and fallback modes.

---

## 16) Compliance Roadmap
- **SOC 2 Type I → Type II** within 12–18 months.
- **GDPR/CCPA** DSR handling, consent, and data mapping. EU region deployment (roadmap).
- **PCI DSS** SAQ-A only (Stripe-hosted payment pages).
- **Security Training** for engineers and support annually.

---

## 17) Secure SDLC
- **Design Reviews:** Threat modeling per epic (STRIDE, LINDDUN for privacy).
- **Code Scanning:** SAST (CodeQL), dependency scans (Dependabot/Snyk), secret scans (gitleaks).
- **PR Gates:** Lint, tests, coverage, schema checks, OpenAPI contract tests.
- **SBOM:** Generated per release; stored with artifacts.
- **Change Management:** Changelog, approvals, rollout plans; canary deploys.

---

## 18) Incident Response
- **Runbooks:** Auth outage, data breach, integration leakage, Polygon unavailability.
- **Severities:** P0 (catastrophic), P1 (major), P2 (moderate), P3 (minor).
- **Process:** Detect → Triage → Contain → Eradicate → Recover → Postmortem (public summary for material incidents).
- **Notifications:** Legal & user comms templates pre-written; regulatory timelines respected.

---

## 19) Anti-Abuse & Acceptable Use
- **Automated Limits:** Rate limiting, anomaly detection on volume and IP patterns.
- **Content Controls:** Journal content not moderated except to prevent abuse (malware links, illegal content); clear AUP.
- **Account Sanctions:** Progressive: warning → restriction → termination.

---

## 20) UI/UX Privacy Notices
- **Just-In-Time** notices for consent (analytics, notifications, embeddings).
- **Privacy Policy** link in onboarding and settings.
- **Cookie Policy:** Minimal cookies; strictly necessary by default; analytics opt-in.

---

## 21) Data Residency & Localization (Roadmap)
- **Regions:** `us-east-1`, `eu-central-1` when enabled; workspace-level residency.
- **Cross-Region Access:** Disabled by default; explicit consent required.

---

## 22) Metrics & KPIs (Security)
- Mean time to detect (MTTD), mean time to resolve (MTTR).
- % of services with current patches.
- % of secrets rotated on schedule.
- # High/Critical vulns open > 30 days (target: 0).

---

## 23) Summary
Traderra’s security and privacy program emphasizes **tenant isolation, least privilege, encryption by default, auditable operations, and privacy-first AI**. Controls are designed for professional users handling sensitive financial records.

> *“Protect the data. Prove the controls.”*
